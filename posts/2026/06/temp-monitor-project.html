分布式温度监测系统——大作业实践笔记 ​

一、项目简介 ​

分布式实时温度监测系统，由多个 STM32 采集节点 + MQTT 云端 + Web 监控大屏组成。

层级技术说明
MCU 固件STM32F103C8T6 + FreeRTOSADC 采集 + MQTT 通信
WiFi 模块ESP8266-01S（AT 指令）串口驱动
后端Python FastAPI + paho-mqttMQTT 订阅、WebSocket 推送、REST API
前端Vue 3 + ECharts + Element Plus实时温度曲线、设备管理、规则配置

四、固件设计 ​

4.1 FreeRTOS 任务划分 ​

任务优先级周期职责
vTaskSensor31500ms（vTaskDelayUntil）NTC 采集 → OLED 显示 → MQTT 上报 → 告警检查
vTaskUrc2100ms 轮询处理 ESP8266 URC 订阅消息
idle0自动FreeRTOS 空闲任务

任务间通过互斥锁保护 OLED（软件 I2C 时序敏感）。

4.2 NTC 温度计算 ​

Steinhart-Hart B 参数方程，注意 ADC 溢出保护（传感器断开时 ADC=4095 会导致除零）：

c
float resistance = SERIES_RESISTANCE * adcValue / (4095.0f - adcValue);
float steinhart = 1.0f / NTC_NOMINAL_TEMPERATURE + logf(resistance / NTC_NOMINAL_RESISTANCE) / NTC_B_COEFFICIENT;
return (1.0f / steinhart) - 273.15f;

4.3 ESP8266 MQTT 通信 ​

使用 AT+MQTTPUBRAW 而非 AT+MQTTPUB——先发长度再发原始字节，避免 JSON 中引号与 AT 指令嵌套冲突。

4.4 URC 消息处理 ​

ESP8266 订阅消息通过 URC（Unsolicited Result Code）主动推送，可能在任意时刻到达。USART1 驱动实现 4 槽 URC 队列，中断中行组装，vTaskUrc 轮询处理。

4.5 报警规则引擎 ​

支持最多 4 条规则，每条包含：type（动作类型）、threshold（阈值）、operator（比较方式）、cooldown_ms（冷却时间）、enabled。

边云告警分离：device_alarm 类型 MCU 本地触发 LED（断网也能工作），email_notify 等类型由 MCU 上报、Python 后端负责发送邮件。

4.6 远程配置链路 ​

Vue 表单 → REST API → MQTT publish → ESP8266 URC → vTaskUrc 解析 → AlarmRule_UpdateFromJson → 即时生效

MCU 端使用 jsmn（零分配 JSON 解析器）+ 自定义点路径访问封装，tokens 数组放 static（约 640 字节），不占堆。

五、Web 后端 ​

5.1 线程安全的 WebSocket 桥接 ​

MQTT 回调在 paho-mqtt 后台线程，WebSocket 推送需在 asyncio 事件循环中执行，使用 asyncio.run_coroutine_threadsafe() 桥接。

5.2 设备状态管理 ​

内存存储，30 秒无数据自动标记离线，每 5 秒检查超时。

5.3 REST API ​

接口方法说明
/api/nodesGET设备列表及状态
/api/nodes/{id}/rulesGET/PUT/DELETE告警规则 CRUD（更新时同步下发到 MCU）

六、前端监控大屏 ​

  • Dashboard.vue：主监控页面（统计卡片 + 温度曲线 + 设备卡片 + 告警日志）
  • RuleConfig.vue：告警规则配置页
  • TempChart.vue：ECharts 多节点实时温度折线图
  • NodeCard.vue：设备状态卡片（温度颜色编码：蓝<30°C、黄30-40°C、红>40°C）

Pinia Store 管理设备状态，WebSocket 自动重连（3 秒延迟），每节点保留最近 100 个温度点。

七、踩坑总结 ​

问题解决方案
AT 指令引号与 JSON 嵌套冲突改用 AT+MQTTPUBRAW，先发长度再发原始字节
URC 消息时序混乱，打断正在发送的指令4 槽 URC 队列 + 中断行组装 + 独立任务轮询
软件 I2C 并发问题（多任务操作 OLED）FreeRTOS 互斥锁 + OLED_ShowStringSafe 封装
vTaskDelay 周期不精确改用 vTaskDelayUntil，从上次唤醒起算
20KB SRAM 无法用 cJSON使用 jsmn（零分配 tokenizer），tokens 放 static

八、开发时间线 ​

阶段天数内容
Phase 1 MVP~6.5 天NTC 采集 → OLED → ESP8266 WiFi → MQTT → Web 基础大屏
Phase 2 扩展~7 天FreeRTOS 多任务 → 报警规则引擎 → 远程配置 → 邮件通知

九、总结 ​

说白了就是 STM32 读温度、WiFi 传上去、网页画个图。技术含量不算高，从 ADC 到浏览器这条线走通了。主要收获是踩了不少坑——AT 指令的引号、串口时序、软件 I2C 抢占，这些都是文档里不会写的。
