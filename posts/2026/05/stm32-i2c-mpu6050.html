STM32软硬件I2C读写MPU6050

STM32软硬件I2C读写MPU6050 ​

来源：江协科技 STM32 入门教程 [10-5] 前置知识：I2C通信协议基础

一、费曼检验（先测后学） ​

1. 一句话说清楚 ​

硬件 I2C 是让 STM32 内置的专用外设模块自动处理 I2C 时序；软件 I2C 是用 GPIO 手动模拟每一个时钟周期和数据位。两者最终都是跟 MPU6050 通过两根线（SCL/SDA）通信，读写它的寄存器。

2. 画出数据流向 ​

┌─────────────┐         I2C 总线          ┌──────────────────────┐
│   MPU6050   │                           │       STM32          │
│             │   SCL (PB6) ──────────►   │                      │
│  传感器数据   │   SDA (PB7) ◄──────────► │  软件I2C: GPIO手动模拟 │
│  (加速度/陀螺) │                           │  硬件I2C: I2C外设自动  │
│             │                           │         │            │
│  寄存器组     │                           │    ┌────▼─────┐      │
│  0x3B~0x48  │                           │    │ DR 寄存器  │      │
└─────────────┘                           │    └────┬─────┘      │
                                          │         │ (CPU/DMA)  │
                                          │    ┌────▼─────┐      │
                                          │    │ 变量/数组  │      │
                                          │    └──────────┘      │
                                          └──────────────────────┘

  软件I2C: CPU 逐 bit 操作 GPIO ──► 每个 bit 都要 CPU 亲自"摆线"
  硬件I2C: CPU 只管往 DR 写数据  ──► 外设自动按协议发出时序

3. 画出硬件关联 ​

  STM32 (STM32F103C8T6)                MPU6050 模块
  ┌────────────────┐                  ┌──────────────┐
  │                │                  │              │
  │  PB6 (I2C1_SCL)├──── SCL ────────┤ SCL      VCC├─── 3.3V
  │                │    ┌──────┐      │              │
  │  PB7 (I2C1_SDA)├────┤4.7kΩ ├───┤ SDA      GND├─── GND
  │                │    └──┬───┘      │              │
  │           GND ─┤──────┘          │  AD0 ────────┤─ GND (地址 0x68)
  │                │                  │              │
  └────────────────┘                  └──────────────┘

  ⚠ 接线两种方式完全相同！区别只在 STM32 内部：
  - 软件I2C：PB6/PB7 配置为 GPIO_Mode_Out_OD（普通开漏输出）
  - 硬件I2C：PB6/PB7 配置为 GPIO_Mode_AF_OD（复用开漏输出）

二、笔记正文 ​

1. 模块概述 ​

项目软件 I2C（模拟）硬件 I2C（外设）
原理GPIO 手动模拟时序STM32 内置 I2C 外设自动处理
所属总线GPIO 挂 APB2I2C1 挂 APB1
时钟来源CPU 主频APB1 时钟
CPU 占用高（每 bit 都要 CPU 参与）低（外设自动收发，可中断/DMA）
引脚任意 GPIO必须用指定引脚（I2C1_SCL=PB6 / I2C1_SDA=PB7）
灵活性高，随便换引脚低，引脚固定
可靠性取决于代码质量硬件保证时序标准
速度受 CPU 主频和代码效率影响由外设时钟决定，更稳定

2. 数据流向详解 ​

数据输入（从 MPU6050 读） ​

阶段软件 I2C硬件 I2C
发起读操作CPU 逐 bit 手动控制 SCL/SDA 电平CPU 写 DR，外设自动产生时序
接收数据CPU 在 SCL 高电平期间手动读 SDA 引脚外设自动采样，数据存入 DR，RXNE 置位
应答控制CPU 手动拉低/释放 SDA设置 ACK 位，外设自动应答

软件 I2C vs 硬件 I2C 数据路径对比 ​

软件 I2C 数据路径：
  CPU ──(逐bit写)──► GPIO_ODR ──► SDA 引脚 ──► MPU6050
  CPU ◄──(逐bit读)── GPIO_IDR ◄── SDA 引脚 ◄── MPU6050
  ⚠ 每个 bit 都要 CPU 参与，占用 100% CPU

硬件 I2C 数据路径：
  CPU ──(写DR一次)──► I2C_DR ──► 移位寄存器 ──► SDA 引脚 ──► MPU6050
  CPU ◄──(读DR一次)── I2C_DR ◄── 移位寄存器 ◄── SDA 引脚 ◄── MPU6050
  ✓ CPU 只在字节级别参与，bit 级操作由硬件完成
  ✓ 可以接 DMA，实现零 CPU 干预传输

3. 硬件电路要点 ​

引脚分配 ​

功能引脚软件 I2C 模式硬件 I2C 模式外部连接
SCLPB6GPIO_Mode_Out_ODGPIO_Mode_AF_ODMPU6050 SCL
SDAPB7GPIO_Mode_Out_ODGPIO_Mode_AF_ODMPU6050 SDA

电气注意事项 ​

注意事项说明
上拉电阻SCL/SDA 必须接上拉电阻（典型 4.7kΩ），总线空闲时为高电平
电压匹配MPU6050 供电 3.3V，STM32 也是 3.3V，无需电平转换
开漏输出必须配置为开漏模式（OD），不能用推挽！I2C 的"线与"特性要求
最大速率标准模式 100kHz，快速模式 400kHz

4. 关键寄存器 ​

硬件 I2C 外设寄存器 ​

寄存器作用常用操作
I2C_CR1控制寄存器 1PE 位使能外设、START/STOP 产生、ACK 使能
I2C_CR2控制寄存器 2设置外设时钟频率
I2C_DR数据寄存器写入=发送，读取=接收
I2C_SR1状态寄存器 1SB、ADDR、TXE、RXNE、BTF 等事件标志
I2C_SR2状态寄存器 2BUSY、MSL 等总线状态
I2C_CCR时钟控制寄存器设置 SCL 频率

5. 初始化流程 ​

硬件 I2C 初始化 ​

c
void MPU6050_Init()
{
    // 步骤 1: 开启时钟
    // 做了什么：同时开启 I2C 外设时钟和 GPIO 时钟
    // 为什么：I2C 挂在 APB1，GPIO 挂在 APB2，两个都要开！
    RCC_APB1PeriphClockCmd(RCC_APB1Periph_I2C1, ENABLE);
    RCC_APB2PeriphClockCmd(RCC_APB2Periph_GPIOB, ENABLE);
    
    // 软件复位I2C2外设
    I2C_SoftwareResetCmd(I2C2, ENABLE);
    I2C_SoftwareResetCmd(I2C2, DISABLE);
    
    // 步骤 2: GPIO 配置
    // 做了什么：PB6(SCL)、PB7(SDA) 配置为复用开漏输出
    // 为什么：硬件 I2C 需要"复用"开漏（AF_OD），不是普通开漏（Out_OD）！
    //         普通开漏 = 软件I2C 用的，复用开漏 = 引脚控制权交给 I2C 外设
    GPIO_InitTypeDef GPIO_InitStructure;
    GPIO_InitStructure.GPIO_Pin = GPIO_Pin_6 | GPIO_Pin_7;
    GPIO_InitStructure.GPIO_Mode = GPIO_Mode_AF_OD;    // ⚠ 复用开漏，不是普通开漏！
    GPIO_InitStructure.GPIO_Speed = GPIO_Speed_50MHz;
    GPIO_Init(GPIOB, &GPIO_InitStructure);
    
    // 步骤 3: I2C 外设配置
    // 做了什么：设置速率、模式、地址模式、ACK 使能
    // 为什么：告诉硬件以什么规则工作
    I2C_InitTypeDef I2C_InitStructure;
    I2C_InitStructure.I2C_ClockSpeed = 100000;                    // 100kHz 标准模式
    I2C_InitStructure.I2C_Mode = I2C_Mode_I2C;                    // I2C 模式（非 SMBus）
    I2C_InitStructure.I2C_DutyCycle = I2C_DutyCycle_2;            // 占空比（高速模式才需要调）
    I2C_InitStructure.I2C_OwnAddress1 = 0xA0;                     // STM32 自身地址（做从机时用）
    I2C_InitStructure.I2C_Ack = I2C_Ack_Enable;                   // 自动应答使能
    I2C_InitStructure.I2C_AcknowledgedAddress = I2C_AcknowledgedAddress_7bit; // 7 位地址模式
    I2C_Init(I2C1, &I2C_InitStructure);
    
    // 步骤 4: 使能 I2C 外设
    // 做了什么：打开 I2C1
    // 为什么：配置完了要"开机"才能用
    I2C_Cmd(I2C1, ENABLE);
}

6. 核心操作（读/写 MPU6050 寄存器） ​

写寄存器流程（硬件 I2C） ​

检查总线空闲 → 产生起始信号 → 等待 EV5（SB 置位）
→ 发送设备地址+W → 等待 EV6（ADDR 置位）
→ 发送寄存器地址 → 等待 EV8（TXE 置位）
→ 发送数据 → 等待 EV8_2（BTF 置位）
→ 产生停止信号

读寄存器流程（硬件 I2C） ​

检查总线空闲 → 产生起始信号 → 等待 EV5（SB 置位）
→ 发送设备地址+W → 等待 EV6（ADDR 置位）
→ 发送寄存器地址 → 等待 EV8（TXE 置位）
→ 产生起始信号（第二次！） → 等待 EV5
→ 发送设备地址+R → 等待 EV6
→ 读取数据（发 NACK 表示只读一个字节）→ 等待 EV7（RXNE 置位）
→ 产生停止信号

⚠ 读操作有两次起始信号！第一次告诉 MPU6050 "我要读哪个寄存器"，第二次切换方向 "现在把数据给我"。

关键事件（EV）速查表 ​

事件含义对应 SR1 标志检查方式
EV5起始信号已发出SB = 1I2C_CheckEvent(I2C1, I2C_EVENT_MASTER_MODE_SELECT)
EV6地址已发送，匹配成功ADDR = 1I2C_CheckEvent(I2C1, I2C_EVENT_MASTER_TRANSMITTER_MODE_SELECTED) 或 I2C_EVENT_MASTER_RECEIVER_MODE_SELECTED
EV8数据寄存器空，可以写入TXE = 1I2C_CheckEvent(I2C1, I2C_EVENT_MASTER_BYTE_TRANSMITTING)
EV8_2数据已移出，传输完成BTF = 1I2C_CheckEvent(I2C1, I2C_EVENT_MASTER_BYTE_TRANSMITTED)
EV7数据寄存器有数据，可读取RXNE = 1I2C_CheckEvent(I2C1, I2C_EVENT_MASTER_BYTE_RECEIVED)

三、踩坑日志 ​

Bug 1：GPIO 时钟开启错误 ​

项目内容
现象I2C 完全不工作，所有读写操作无响应
排查过程示波器看 SCL/SDA 引脚，发现完全没有波形
根本原因复制代码后忘记改时钟，错误开启了 GPIOA 时钟，实际用的是 GPIOB
解决方案RCC_APB2PeriphClockCmd(RCC_APB2Periph_GPIOB, ENABLE) 改正即可
预防措施配置 GPIO 时先检查引脚属于哪个端口，再查该端口挂在哪条总线

教训： 最低级的错误往往最难发现——因为"代码能编译能运行"，但引脚就是不工作。以后遇到"引脚没反应"，第一个查时钟。

Bug 2：时序错误 — SCL 循环结束后未拉低 ​

项目内容
现象读取 MPU6050 数据不稳定，偶发通信失败
排查过程逐函数检查时序，发现 I2C_Write() 和 I2C_Read() 的 8 位循环结束后 SCL 停留在高电平
根本原因字节发送/接收完毕后，没有把 SCL 拉低就进入下一个操作（ACK 或下一字节），导致时序歧义
解决方案循环结束后补一行 I2C_Write_SCL(0);，确保 SCL 在低电平状态交出控制权
预防措施每个时序函数结束时，检查 SCL 是否处于"安全态"（低电平）

Bug 3：硬件I2C时钟未开启 ​

项目内容
现象I2C 完全不工作，所有读写操作无响应
排查过程MPU6050_Init()内写寄存器0x6B过程阻塞
根本原因忘记开启I2C外设时钟（APB1总线）
解决方案RCC_APB1PeriphClockCmd(RCC_APB1Periph_I2C2, ENABLE)
预防措施先看一遍结构图

Bug 4：读数据时序错误 ​

项目内容
现象读出数据错误
排查过程尝试读出who am i标志位发现不为68，怀疑读接收字节部分有错误
根本原因时序错误先读了数据再等事件完成 data[i] = I2C_ReceiveData(I2C2); while (!I2C_CheckEvent(I2C2, I2C_EVENT_MASTER_BYTE_RECEIVED));
解决方案先等事件完成： while (!I2C_CheckEvent(I2C2, I2C_EVENT_MASTER_BYTE_RECEIVED)); 再读数据：data[i] = I2C_ReceiveData(I2C2);
预防措施先看清手册描述

四、软件 I2C vs 硬件 I2C 详细对比 ​

对比项软件 I2C硬件 I2C
GPIO 模式GPIO_Mode_Out_OD（普通开漏）GPIO_Mode_AF_OD（复用开漏）
时钟来源无（CPU 逐 bit 手动控制）APB1 时钟驱动
代码复杂度需要手动写每个时序函数配置寄存器后调库函数
调试难度看波形就知道哪里错要理解 EV 事件状态机
最大速率取决于代码效率，一般较慢硬件保证，可达 400kHz
CPU 占用100%（每个 bit 都要管）极低（可接 DMA）
换引脚随便换只能用固定引脚
适用场景引脚不够用、需要灵活分配追求性能、标准项目开发

五、轮询 vs 中断 vs DMA 对比 ​

方式优点缺点适用场景
轮询简单直观，容易调试CPU 空等，浪费资源学习阶段、简单读写
中断CPU 可以做其他事状态机复杂，调试困难需要实时性的系统
DMA几乎零 CPU 开销配置复杂，需要 DMA 通道高速连续采集（如读 14 字节原始数据）
