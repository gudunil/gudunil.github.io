STM32 I2C通信协议详解 ​

来源：江协科技 STM32 入门教程 [10-1]

一、概述 ​

  • I2C（Inter-Integrated Circuit），两根线实现半双工通信
    • SCL：时钟线（Serial Clock）
    • SDA：数据线（Serial Data）
  • 同步、半双工、多主多从总线
  • 通过设备地址区分不同从机

二、时序基本单元 ​

信号条件说明
起始条件SCL 高电平时，SDA 从高→低主机发出，总线开始
停止条件SCL 高电平时，SDA 从低→高主机发出，总线释放
发送一个字节SCL 低电平期间改变 SDA，高电平期间采样MSB 先发（高位在前）
接收一个字节同上，但从机控制 SDA主机回复 ACK/NACK
发送 ACKSCL 高电平期间，SDA = 0接收方确认收到
发送 NACKSCL 高电平期间，SDA = 1接收方不想再收了

三、读写时序 ​

写操作（主机 → 从机） ​

[S] [地址+W] [ACK] [寄存器地址] [ACK] [数据] [ACK] ... [P]

读操作（从机 → 主机） ​

这里想读多个字节就在ACK时发送，应答0（会在最开始指定的寄存器往后增长），不想读了就发送非应答1，再stop就可以了

[S] [地址+W] [ACK] [寄存器地址] [ACK]
[S] [地址+R] [ACK] [数据] [ACK] ... [NACK] [P]

注意读操作有两次起始信号：第一次指定要读哪个寄存器，第二次才切换方向读数据。

四、硬件电路 ​

  • SCL 和 SDA 都接上拉电阻（默认高电平）
  • 设备输出低电平 = 拉低，释放 = 上拉恢复高电平
  • 这种线与特性允许多个设备共享总线

五、踩坑日志 ​

Bug 1：GPIO时钟开启错误 ​

这里我把其他部分开启时钟的代码复制过来以后却忘记了更改，错误开启了GPIOA的时钟，导致后续所有操作都没用，这是最严重也是最低级的错误。

c
void MyI2C_Init(void)
{
    //错误部分，已改正为GPIOB
    RCC_APB2PeriphClockCmd(RCC_APB2Periph_GPIOB, ENABLE);

    GPIO_Init(I2C_GPIO_PORT, &(GPIO_InitTypeDef){
        .GPIO_Pin = I2C_SCL_PIN | I2C_SDA_PIN,
        .GPIO_Mode = GPIO_Mode_Out_OD,
        .GPIO_Speed = GPIO_Speed_50MHz});
}

Bug 2：时序错误：I2C_Write() 和 I2C_Read() 循环结束时的SCL状态 ​

这里我总在把SCL拉高后，忘记给SCl拉低让其为后续操作做准备。具体表现为：在I2C_Write 和 I2C_Read 的8位循环结束后，SCL都停留在了高电平。

SCL没有拉低的影响：

  • 字节传输完毕后，应该主动拉低SCL，把总线控制权交出，或者让总线处于空闲前的准备状态。
  • 更重要的是，在 I2C_Read 中，如果循环结束时SCL为高，且你没有及时释放SDA（虽然你在开头释放了），如果下一刻要发送ACK，容易产生时序歧义。
c
uint8_t I2C_WaitAck(void)
{
    // 主机在接收之前，需要释放SDA
    I2C_Write_SDA(1);
    // 拉低SCL，让从机在SDA上放ACK
    I2C_Write_SCL(0);
    // 拉高SCL，主机读取SDA上的ACK
    I2C_Write_SCL(1);
    uint8_t ack = I2C_Read_SDA();
    
    //错误部分：没有将SCL电平拉低走完完整时序,已添加拉低SCL代码
    I2C_Write_SCL(0);
    return ack;
}

Bug 总结 ​

项目内容
现象I2C 完全不工作，所有读写操作无响应
排查过程示波器看 SCL/SDA 引脚，发现完全没有波形
根本原因复制代码后忘记改时钟，错误开启了 GPIOA 时钟，实际用的是 GPIOB
解决方案RCC_APB2PeriphClockCmd(RCC_APB2Periph_GPIOB, ENABLE) 改正即可
预防措施配置 GPIO 时先检查引脚属于哪个端口，再查该端口挂在哪条总线

教训： 最低级的错误往往最难发现——因为"代码能编译能运行"，但引脚就是不工作。以后遇到"引脚没反应"，第一个查时钟。

项目内容
现象读取 MPU6050 数据不稳定，偶发通信失败
排查过程逐函数检查时序，发现 I2C_Write() 和 I2C_Read() 的 8 位循环结束后 SCL 停留在高电平
根本原因字节发送/接收完毕后，没有把 SCL 拉低就进入下一个操作（ACK 或下一字节），导致时序歧义
解决方案循环结束后补一行 I2C_Write_SCL(0);，确保 SCL 在低电平状态交出控制权
预防措施每个时序函数结束时，检查 SCL 是否处于"安全态"（低电平）

错误代码 vs 正确代码：

c
// ⚠ 原代码：WaitAck 中 SCL 没拉低
uint8_t I2C_WaitAck(void)
{
    I2C_Write_SDA(1);       // 释放 SDA
    I2C_Write_SCL(0);
    I2C_Write_SCL(1);
    uint8_t ack = I2C_Read_SDA();
    // ❌ 这里缺了 I2C_Write_SCL(0);
    return ack;
}

// ✓ 修正后
uint8_t I2C_WaitAck(void)
{
    I2C_Write_SDA(1);
    I2C_Write_SCL(0);
    I2C_Write_SCL(1);
    uint8_t ack = I2C_Read_SDA();
    I2C_Write_SCL(0);       // ✅ 拉低 SCL，走完完整时序
    return ack;
}

教训： I2C 时序是"一环扣一环"的，每个函数出口都要保证 SCL 处于低电平，否则下一拍就会出问题。养成习惯：函数末尾，SCL 必低。
